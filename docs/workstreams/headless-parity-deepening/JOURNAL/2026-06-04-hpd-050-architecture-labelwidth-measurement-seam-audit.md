# HPD-050 - Architecture LabelWidth Measurement Seam Audit

Date: 2026-06-04
Task: HPD-050 Architecture-first layout engine audit

## Context

The previous Mermaid/Cytoscape source audit narrowed the next production-capable Architecture
seam to browser-faithful service `labelWidth`. The open question was whether the repository already
had a reusable headless-shell or browser text lookup mechanism that Architecture could consume
directly, rather than adding another diagram-specific patch.

## Findings

- The shared `TextMeasurer` trait already separates several SVG text probes:
  `measure_svg_text_bbox_x(...)`, `measure_svg_simple_text_bbox_width_px(...)`, computed text
  length, and wrapped text measurement. This is the right renderer-side abstraction for
  deterministic SVG text and root-bounds estimates.
- Existing generated text override infrastructure is intentionally scoped by diagram and phase.
  `text/overrides.rs` exposes Flowchart and Sequence generated lookup helpers, while C4 owns its
  own lookup from `c4.rs` through `generated/c4_text_overrides_11_12_2.rs`.
- The C4 headless-shell table is not a generic browser measurement backend. It is generated by
  `xtask gen-c4-text-overrides` from upstream C4 SVG `<text>` nodes and Puppeteer
  `headless: "shell"` `getBBox().width` probes. It is keyed by normalized font family, font size,
  font weight, and exact text, and is classified by `report-overrides` as fixture/browser text
  lookup data.
- Architecture service `labelWidth` is a different upstream phase from C4. The relevant value is
  Cytoscape renderer label measurement for compound child sizing, not SVG `createText(...)`
  `getBBox()`. Cytoscape label bounds then add `marginOfError = 2` around that stored
  `labelWidth`.
- The current Architecture probe infrastructure already captures the browser side needed for this
  phase:
  - `xtask debug-architecture-fcose-probe` wraps
    `tools/debug/arch_fcose_browser_probe_fixture_025.js` and writes stable
    `*.fcose-browser-probe.json` / Markdown artifacts.
  - The raw JSON includes final service `metrics.labelWidth` / `metrics.labelHeight`,
    `labelBounds.all`, `bodyBounds`, and final `node.boundingBox()`.
  - `xtask debug-architecture-delta --probe-dir` already consumes those probe artifacts and joins
    browser `metrics.labelWidth` with local `ArchitectureCytoscapeServiceBounds.label_metrics`.
- The active seven-probe batch under
  `target\compare\architecture-fcose-probe-label-contribution-active-residuals-hpd050` contains
  service `labelWidth` evidence for the current residual set:
  - `batch5`: 4 service label widths, first `runner=149`
  - `batch6_init_fontsize_icon_size_wrap`: 3, first `api=95`
  - `group_port_edges`: 4, first `in1=23`
  - `html_titles`: 3, first `web=123`
  - `junction_fork_join`: 5, first `ingress=53`
  - `nested_groups`: 5, first `ingress=53`
  - `unicode`: 4, first `metrics=117`

## Outcome

No production behavior changed.

The reusable part is the probe/report pipeline, not the C4 generated table. The repository already
has a stable way to collect and join browser-faithful Architecture `labelWidth` evidence, but it
does not yet have a reusable generated Architecture `labelWidth` lookup backend. Adding such a
lookup would be mechanically possible, but previous exact lookup experiments already showed that
`labelWidth` alone leaves the final group phase half-modeled and can increase the full Architecture
root queue.

The next implementation candidate should therefore pair any browser-faithful label measurement with
the source child-union and final `node.boundingBox()` expansion phases. A lookup-only production
patch remains rejected.
